6. Бэкапы и аварийное восстановление

RTO: 4 часа RPO: 6 часов Трёхуровневая стратегия бэкапов с off-site хранением.

🗃 PostgreSQL
PostgreSQL (live) → WAL архивация (непрерывно) → pg_dump каждые 6ч → /backup/pg/ (локальный диск) → Backblaze B2 (ежедневно)
Размер: ~5-20 ГБ
Retention: 30 дней локально, 90 дней B2
Восстановление: pg_restore, ~15-30 мин
📦 MinIO (прошивки + артефакты)
MinIO buckets → rsync incremental (по событию) → /backup/firmware/ (SATA SSD) → Backblaze B2 (еженедельно)
Размер: 500 ГБ — 4 ТБ
Retention: Навсегда (прошивки уникальны)
Восстановление: rsync обратно, 1-4 часа
📄 Конфигурации
docker-compose.yml nginx.conf .env, скрипты → git commit (при изменении) → GitHub private repo
Размер: < 10 МБ
Retention: Полная git-история
Восстановление: git clone, 1 мин
⚠️ Disaster Recovery (план)
1. Установить Ubuntu на новый сервер (~30 мин) 2. git clone конфигов (~1 мин) 3. docker compose up -d (~10 мин) 4. pg_restore из B2 (~30 мин) 5. rsync firmware из B2 (~2-3 часа) 6. Проверка + DNS switch (~15 мин)
RTO: 4 часа (полное восстановление)
RPO: 6 часов (потеря данных максимум)
Тестирование: ежемесячный dry-run

Визуальная схема потоков бэкапа

PostgreSQL ──WAL──► /backup/pg/wal/ ──daily──► Backblaze B2 │ ($5/ТБ/мес) └──pg_dump/6h──► /backup/pg/dumps/ ──daily──► │ │ MinIO ──rsync──► /backup/firmware/ ──weekly─► │ (прошивки) (SATA SSD, 8 ТБ) │ │ Configs ──git push──► GitHub Private Repo │ (docker,nginx) (полная история) │ │ Redis ──AOF/RDB──► /backup/redis/ ──daily──► ▼ Backblaze B2 (Ташкент → US-West) encryption: AES-256 at rest retention: 90 дней

Правила алертов

🚨 Critical (немедленно)

Сервис не отвечает > 2 мин

Диск заполнен > 95%

RAM > 95% + OOM Kill

Backup не выполнился > 24ч

PostgreSQL replica lag > 1 мин

⚠️ Warning (в рабочее время)

Диск > 85%

RAM > 85%

Task queue > 20 задач

Воркер idle > 1 часа

Скилл error rate > 30%

Воркер повторяет вопрос 3+ раз
