0. Асинхронность и мульти-агентность

Архитектурный принцип #1 Воркер никогда не ждёт. Пока одна задача выполняется на сервере, он работает над следующей. Система параллелизирует всё что можно.

Проблема без асинхронности: jadx декомпилирует APK = 5 мин. Ghidra анализирует бинарь = 30 мин. apk-rebuild = 2 мин. Воркер сидит и ждёт. При 23 APK × 7 минут = 2.5 часа чистого ожидания в день. При 6 воркерах = 15 человеко-часов потерь ежедневно.

Принцип: Fire-and-Forget + Notification

Воркер: "декомпилируй com.deepal.hvac" │ ▼ MCP: task_id=abc123, status=queued ← мгновенный ответ (100ms) │ ▼ Воркер: "пока жду, переведи strings.xml ← уже работает над другим от com.deepal.media" │ ▼ MCP: task_id=def456, status=queued │ ...5 минут проходит... │ ▼ Сервер → Claude Code: "abc123 готов! ← push-нотификация com.deepal.hvac декомпилирован, найдено 42 строки, 3 хардкод в smali" │ ▼ Воркер продолжает с hvac, media ещё собирается в фоне

3 уровня параллелизма

Уровень 1: Параллельные задачи одного воркера

Воркер может иметь до 5 фоновых задач одновременно. Пока jadx декомпилирует APK #1, он уже переводит APK #2 (который был декомпилирован раньше), а APK #3 собирается в фоне после перевода.

Конвейер: decode → translate → rebuild → install идут не последовательно для всех APK, а конвейером: APK #1 на этапе rebuild, APK #2 на translate, APK #3 на decode — одновременно.

Уровень 2: Мульти-агент внутри скилла

Скилл voice-setup (Level 3) запускает 3 суб-агента параллельно:

Агент 1: сканирует разрешения через ADB
Агент 2: ищет API функций авто в декомпилированном коде
Агент 3: подготавливает шаблон патча фреймворка

Все три работают одновременно. Оркестратор собирает результаты и передаёт дальше. Экономия: 15 мин вместо 45 мин последовательно.

Уровень 3: Спекулятивное выполнение

Система предугадывает что понадобится дальше и запускает заранее:

Воркер начал переводить APK → система в фоне уже:
• Декомпилирует следующие 3 APK по приоритету
• Извлекает строки из них
• Ищет похожие записи в базе знаний
• Готовит предварительный перевод автомобильных терминов

Когда воркер закончит текущий APK — следующий уже готов к работе.

Паттерн нотификации

Как Claude Code узнаёт что задача готова?

Вариант A (рекомендуемый): MCP Notifications. MCP-протокол поддерживает server → client notifications. Сервер отправляет notifications/task_complete с результатом. Claude Code получает и вплетает в контекст разговора.

Вариант B (fallback): Smart polling. Tool check_tasks() проверяет все фоновые задачи воркера. Claude Code вызывает его периодически между другими действиями. Не отдельным сообщением — а попутно, как часть workflow.

Вариант C (для критичных): Webhook → Telegram. Длинные задачи (Ghidra 30 мин) → бот в Telegram: «Ghidra анализ firmware.bin завершён. 47 функций найдено. Продолжить в Claude Code.» Воркер мог уйти на обед.

Конвейерный перевод 23 APK

Без конвейера (последовательно): APK1: [decode 2m][translate 5m][rebuild 2m] = 9m APK2: [decode 2m][translate 5m][rebuild 2m] = 9m APK3: [decode]... Итого 23 APK: ~207 минут = 3.5 часа С конвейером (параллельно): APK1: [decode 2m][translate 5m][rebuild 2m] APK2: [decode 2m][translate 5m][rebuild 2m] APK3: [decode 2m][translate 5m][rebuild 2m] APK4: [decode 2m]... Итого 23 APK: ~70 минут = 1.2 часа (3x быстрее) С конвейером + спекулятивное: APK1-5: [pre-decoded]→[translate 5m][rebuild 2m] APK6-10: [decoding...]→[translate 5m]... Итого: ~50 минут (4x быстрее)

Главный принцип: Сервер всегда работает. Воркер всегда работает. Никто никого не ждёт. Задачи текут как конвейер, длинные уходят в фон, система предугадывает следующие шаги. Это не оптимизация — это фундамент архитектуры.

1. Архитектура данных

Ключевая сложность Один дамп прошивки = 8-64 ГБ. 50 автомобилей = 2-5 ТБ сырых данных, до 15 ТБ с распакованными и декомпилированными артефактами.

Двухуровневое хранилище

MinIO (S3-совместимый) — для всего тяжёлого

Сырые дампы, образы партиций, архивы декомпиляции. Дедупликация через content-addressable storage: хеширование блоков по 4 МБ. Одинаковые vendor-партиции Deepal S07 и L07 (70-85% совпадение) хранятся один раз.

XFS файловая система — рабочая область

Активные проекты: текущая декомпиляция, файлы под анализом. XFS оптимальна для миллионов мелких файлов (smali). Scratch-раздел минимум 128 ГБ на NVMe.

Пайплайн распаковки прошивки

simg2img super.img super.raw          # 2 мин на 8 ГБ
    ↓
lpunpack super.raw ./partitions/      # параллельно
    ↓
mount -o ro,loop → извлечение APK     # параллельно по партициям
    ↓
parallel -j$(nproc) apktool d {} :::  # полная параллелизация
    ↓
Индексация строк → полнотекстовый поиск

Хранение декомпилированных APK

Один APK = 500-5000 файлов. 200+ APK на прошивку × 20 машин = миллионы файлов. Решение: хранить как tar.zst архивы в MinIO (smali сжимается в 10-15 раз). Распаковывать только для активного анализа.

Удалённый доступ через MCP

Воркеры НЕ скачивают гигабайты. MCP-инструменты работают на сервере, клиенту уходит только результат: текст, diff, список файлов. Трафик — килобайты вместо гигабайт.

Сравнение прошивок (diff)

Трёхуровневый подход: (1) хеши партиций — мгновенно, (2) файловый diff между образами — секунды, (3) diff smali-кода изменённых APK — минуты. Результаты кешируются в SQLite.

2. MCP-сервер и движок скиллов

Ядро системы Все инструменты, вся логика, вся защита IP — в одном серверном компоненте.

Архитектура

Claude Code (worker) ──MCP/SSE──► MCP Server (Python, mcp SDK) │ FastAPI (REST + WS) │ ┌──────────┼──────────┐ ▼ ▼ ▼ Redis PostgreSQL MinIO (очередь, (скиллы, (артефакты: сессии, история, APK, дампы, блокировки) воркеры) результаты) │ Celery Workers (N=4) │ ┌─────────┼─────────┐ ▼ ▼ ▼ Docker Docker Docker (apktool) (jadx) (ghidra) │ Garage Server (ADB bridge, scrcpy, WireGuard)

Проблема длительных операций

MCP-протокол ожидает синхронный ответ. Декомпиляция — 3-10 минут. Ghidra — до 30 минут.

Решение: submit + poll

Тяжёлый tool мгновенно возвращает {"task_id": "abc123", "status": "queued"}. Отдельный tool task_status(id) показывает прогресс. Claude Code сам решает когда проверить — это естественно для LLM-агента.

Concurrency = 4. Один воркер — максимум 2 тяжёлые задачи. Если очередь > 8 — новые отклоняются с сообщением.

Мульти-агентная оркестрация (Level 3)

Серверный оркестратор на Python. Каждый суб-шаг получает только свой срез контекста, а не всю историю. Между шагами — compressed state: не сырой вывод jadx (100 МБ), а structured summary. Для LLM-шагов — Claude API напрямую с сервера.

Hot-reload скиллов

Скиллы хранятся как данные в PostgreSQL (prompt, tool_config, версия). MCP читает текущую версию при каждом вызове (кеш 30 сек). Rollback = переключение is_active. A/B тестирование: часть воркеров на версии A, часть на B.

3. База знаний и самообучение

Конкурентное преимущество С каждым проектом система умнеет. 10-й похожий автомобиль — почти автоматически.

Гибридная модель данных

Структурированное ядро (PostgreSQL)

Таблицы: vehicles, apk_entries, access_methods, permissions, endpoints. Точные SQL-запросы: «все APK с хардкод на api.changan.com».

JSONB документы

Для знаний без фиксированной схемы. GIN-индексы для быстрого поиска по вложенным полям.

Векторные эмбеддинги (pgvector)

Паспорт автомобиля → эмбеддинг. При поступлении L07 → cosine similarity с S07 = 87%. HNSW-индекс, поиск < 5 мс на 15 000 записей.

Граф связей (adjacency list)

APK → Service → Permission → System. 3 уровня глубины. Рекурсивные CTE в PostgreSQL. Не Neo4j — объём мал, не оправдывает отдельный сервис.

Автоматическое извлечение знаний

Канал A: Structured output от MCP-скиллов — каждый tool возвращает JSON с метаданными. Канал B: Post-session hook — анализ файлов, извлечение URL, permissions. Канал C: Еженедельная суммаризация сессий через Claude Haiku.

Кросс-проектный трансфер

Новый проект → эмбеддинг метаданных → top-3 похожих проекта → для каждой записи вычисляется применимость (high/medium/low) → «стартовый пакет» из 50-80 релевантных записей с пометками что перепроверить.

Устаревание знаний

Confidence score: ручной ввод = 0.95, автоизвлечение = 0.7, трансфер = 0.5. Decay: при новой версии прошивки confidence *= 0.8. Через 3 версии без подтверждения → флаг «требует ревалидации».

Стек: PostgreSQL 16 + pgvector + JSONB. Одна БД, три типа индексов. Никакого Neo4j, никакого отдельного vector store. Простота — конкурентное преимущество.

4. Безопасность и защита IP

Критично Работник использует инструменты, но никогда не видит исходный код, промпты или базу знаний.

5 уровней защиты

УровеньЧто защищаетКак
1. Серверное исполнениеВесь код инструментовКод не покидает сервер. Worker видит только API.
2. Prompt defenseСистемные промптыКритическая логика в MCP-инструментах, не в промптах. Injection defense в системном промпте.
3. MCP opacityРеализация toolsМинимальные описания. Агрегированные инструменты вместо гранулярных. Результат как ссылка, не файл.
4. Токены + RBACДоступ к даннымJWT с TTL 15 мин. Привязка к fingerprint машины. Per-project ACL.
5. Аудит + revokeУтечки и увольненияAppend-only WORM лог. Мгновенный revoke при увольнении. Корпоративные устройства с MDM.

Защита алгоритмов (Level 4 скиллы)

Полностью скрыть алгоритм от опытного реверсера невозможно. Задача — сделать анализ настолько дорогим, чтобы он превышал экономическую выгоду от кражи.

Полиморфизм: каждая сборка использует разные имена классов, разные точки встраивания. Многоуровневость: 15-20 проверок с разными механизмами (CRC, hash, signature, timing). Native: критические проверки в .so с anti-debug. Серверная валидация: часть проверок требует обращения к серверу активации.

Offboarding

Технически: мгновенный revoke токенов, удаление контейнера, стирание данных. Организационно: NDA с финансовыми последствиями. Ротация алгоритмов защиты после увольнения ключевого сотрудника. Корпоративные устройства с remote wipe.

Docker Compose — достаточно

Бэкапы

PostgreSQL: pg_dump каждые 6ч + WAL. Прошивки: rsync incremental при поступлении. Артефакты: restic ежедневно. Off-site: Backblaze B2 (500 ГБ = $2.50/мес). RTO: 4 часа.

6. Аналитика и мониторинг

Обратная связь Без аналитики нет улучшения скиллов, нет понимания эффективности команды.

Что собирать

Каждый MCP-вызов (инструмент, параметры, длительность, успех/провал, токены). Каждое сообщение воркера (текст для анализа вопросов). Каждая ADB-команда. Файловые операции. Сессионные события (начало/конец, idle).

Анализ вопросов — 3 уровня

Уровень 1: Эвристики

«Что такое X» = пробел в знаниях. «Не работает» = проблема скилла. Regex, мгновенно, бесплатно.

Уровень 2: Кластеризация

Embeddings + HDBSCAN раз в сутки. Растущий кластер = системная проблема.

Уровень 3: LLM-классификация

Claude Haiku для неизвестных вопросов. Категория, серьёзность, рекомендация. ~$0.5/день на 500 вопросов.

Движок алертов

SQL-запросы на скользящих окнах. Проверка каждые 30 сек. Примеры: воркер задаёт похожий вопрос 3+ раз за 2 часа → алерт админу. Скилл падает >30% → алерт. Воркер idle >1 час → алерт лиду.

Стоимость токенов

Перевод одного APK среднего размера: $2-8 в токенах. Полный проект автомобиля (20+ APK + всё остальное): $50-200. Трекинг по воркеру, скиллу, проекту, клиенту.

7b. Мультиязычность

Мультипликатор дохода Одна прошивка × N языков = N продуктов. Перевод на арабский после русского занимает 30% времени первого перевода благодаря переиспользованию.

Архитектура локализации

Один проект = много языков

При создании проекта выбираются целевые языки. Каждый язык — отдельный трек с собственным прогрессом, тестированием и quality gate. Но переводы переиспользуются: автомобильные термины, технические строки, layout-структуры.

Эффект переиспользования

Первый язык (русский): 100% работы = базовый трек Второй язык (арабский): 30% работы + RTL адаптация Третий язык (испанский): 25% работы (RTL не нужен) Четвёртый язык (узбекский): 20% работы (похож на русский) Причина экономии: ├── Smali хардкод: найден один раз — патчится для всех языков ├── Серверные строки: структура одинакова, меняется только текст ├── Виджеты: layout готов, меняются strings ├── ADB/Root/Защита: не зависят от языка (делаются один раз) └── База знаний: 80% записей language-agnostic

RTL (арабский, иврит) — отдельная сложность

RTL (Right-to-Left) требует зеркалирования UI: навигация справа налево, прогресс-бары инвертированы, layout_direction=rtl. Это затрагивает каждый layout XML в каждом APK. Нужен отдельный скилл rtl-adapter для автоматической адаптации.

Архитектура работы Каждая задача = сессия. Сессия имеет лог, результат, возможность зайти и продолжить. Результат каждой сессии — pull request в головной скилл.

Жизненный цикл сессии

1. Создание

Система создаёт сессию автоматически при начале задачи. Или воркер создаёт вручную: claude --session "Перевод hvac AR". Сессия привязана к проекту, этапу, языку.

2. Работа

Воркер работает в Claude Code как обычно. Все сообщения, tool calls, результаты логируются. Сессию можно приостановить и вернуться — контекст сохраняется.

3. Вмешательство

Админ или тимлид может войти в любую сессию через CLI:

claude --join-session SESSION_ID

Видит весь лог, может добавить сообщения, вызвать инструменты, исправить ошибки. Его действия логируются отдельно: [admin] Улугбек: исправил RTL direction.

4. Результат → Pull Request

Когда задача готова, сессия формирует PR в головной скилл:

Тимлид смотрит PR в админке или CLI. Может: одобрить (merge), запросить изменения (request changes), или войти в сессию и исправить сам. После мержа — результат применяется к финальной сборке.

7d. Версионирование скиллов

Матричное версионирование Один скилл может работать по-разному на SA8155P vs SA8295P, Android 10 vs 12, прошивке v2.1 vs v3.0.

Проблема

Скилл apk-translate работает одинаково для всех. Но root-helper — совершенно разный для SA8155P (fastboot) и SA8295P (EDL mode). voice-setup отличается между Android 10 (старый AudioManager) и Android 12 (новый CarAudioManager). adb-finder зависит от конкретной версии прошивки OEM.

Матрица версий скилла

root-helper ├── v1.0 [SA8155P, Android 10-11] │ └── Magisk через fastboot boot ├── v1.1 [SA8155P, Android 12] │ └── Magisk через fastboot boot + AVB disable ├── v2.0 [SA8295P, Android 12] │ └── EDL mode → patch boot → flash └── v2.1 [SA8295P, Android 13] └── EDL mode → новый формат super partition voice-setup ├── v1.0 [Android 10, все процессоры] │ └── AudioManager + MediaRecorder ├── v2.0 [Android 12+, SA8155P] │ └── CarAudioManager + AudioRecord └── v2.1 [Android 12+, SA8295P] └── CarAudioManager + кастомный HAL adb-finder ├── v1.0 [Deepal, прошивка ≤ 2.x] │ └── diag port + diag_enter_adb ├── v1.1 [Deepal, прошивка 3.x] │ └── diag port закрыт, через engineering menu ├── v2.0 [Changan, все прошивки] │ └── Климат+громкость 5сек → инж. меню └── v3.0 [Generic fallback] └── Поиск через nmap + known exploits

Как это работает в PostgreSQL

skills (
  id, name, version,
  soc         TEXT[],    -- ['SA8155P','SA8295P'] или NULL = все
  android_ver TEXT[],    -- ['10','11','12'] или NULL = все
  oem         TEXT[],    -- ['deepal','changan'] или NULL = все
  fw_version  TEXT[],    -- ['2.x','3.x'] или NULL = все
  prompt, tool_config, is_active, created_at
)

-- Запрос: найти подходящий скилл для SA8295P + Android 12 + Deepal
SELECT * FROM skills
WHERE name = 'root-helper'
  AND (soc IS NULL OR 'SA8295P' = ANY(soc))
  AND (android_ver IS NULL OR '12' = ANY(android_ver))
  AND (oem IS NULL OR 'deepal' = ANY(oem))
  AND is_active = true
ORDER BY version DESC LIMIT 1;
-- → v2.0 [SA8295P, Android 12]

Автоматический выбор версии

При старте проекта система определяет платформу (SoC, Android, OEM, прошивка) из дампа. Все скиллы автоматически резолвятся на нужную версию. Воркер не думает об этом — он просто говорит «найди ADB», а система выбирает правильный adb-finder v1.1 для Deepal прошивка 3.x.

Fallback-цепочка

Поиск скилла: root-helper для SA8295P + Android 13 + Deepal + fw 4.0 │ ├── Точное совпадение: SA8295P + Android 13 + Deepal + fw 4.0 → НЕТ ├── Без прошивки: SA8295P + Android 13 + Deepal → НЕТ ├── Без OEM: SA8295P + Android 13 → v2.1 ✓ НАЙДЕН │ └── Используем v2.1, но помечаем: ⚠️ "Версия не проверена для прошивки 4.0 Deepal" ⚠️ "Результат требует ручной проверки"
Главное: скилл — не один файл, а матрица реализаций. Система автоматически выбирает правильную. Новая платформа — добавляется новая строка в матрицу, остальные продолжают работать. Это позволяет масштабировать на десятки моделей автомобилей без переписывания скиллов.
